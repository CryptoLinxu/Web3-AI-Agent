# Web3 AI Agent PRD MVP

## 1. 产品背景与转型目标

本项目服务于个人转型目标：从 `Web3 前端工程师` 升级为 `AI 应用工程师 / Agent 工程师`。项目既是学习载体，也是未来可展示的作品集基础。

产品层面，本项目要验证的不是“做一个聊天页面”，而是“做一个能够理解用户意图、调用 Web3 工具、返回可信结果，并具备最小风险边界的 AI Agent”。

---

## 2. 产品目标

### 核心目标

- 构建一个可运行的 Web3 AI Agent MVP
- 覆盖 `对话 + Tool Calling + Agent Loop + 最小 Memory` 四个核心能力
- 形成一套文档先行、分阶段推进、适合 vibe coding 的开发体系

### 非技术目标

- 为后续求职、作品展示和能力证明提供核心项目
- 建立一套后续可复用的 AI Agent 开发方法论

---

## 3. 目标用户画像

### 用户画像 A：Web3 普通用户

- 有钱包地址，但不熟悉链上数据查询
- 希望通过自然语言快速获取余额、价格、Gas 等信息
- 不希望阅读复杂区块浏览器页面

### 用户画像 B：转型中的开发者本人

- 需要一个可持续迭代的项目作为学习主线
- 需要用项目反向组织对 Agent 的理解
- 需要形成文档、架构、测试与实现的完整闭环

---

## 4. 核心使用场景

### 场景 1：查询实时价格

用户输入：“ETH 现在价格是多少？”

期望：

- Agent 能识别问题属于价格查询
- 自动调用价格工具
- 返回结构化但易懂的结果

### 场景 2：查询地址余额

用户输入：“帮我查一下这个地址的 ETH 余额：0x...”

期望：

- Agent 能识别钱包地址
- 调用链上余额工具
- 返回余额结果并说明数据来源

### 场景 3：多轮跟进

用户先问价格，再问：“如果是我刚才那个地址呢？”

期望：

- 系统能保留当前对话上下文
- 在合理范围内复用已有上下文

### 场景 4：风险边界处理

用户输入：“你帮我判断现在该不该重仓买 ETH”

期望：

- 系统不直接给出交易建议
- 提供数据参考、风险提示与免责声明

---

## 5. MVP 功能范围

### 必做范围

- 基础聊天界面
- 流式输出
- Tool Calling
- Agent Loop v1
- 最小会话 Memory
- Web3 工具：
  - `getETHPrice`
  - `getWalletBalance`
  - `getGasPrice` 或 `getTokenInfo` 二选一
- 错误处理与降级回复
- 风险提示与免责声明

### 文档范围

- 学习资料清单
- 项目里程碑 checklist
- PRD
- 阶段执行说明
- skill 方案文档

---

## 6. 非目标

以下能力不属于 MVP：

- 自动交易执行
- 钱包真实签名与链上写操作
- 多链复杂资产聚合
- 高级 RAG 知识库系统
- 长期持久化 Memory
- 多 Agent 协作
- 完整后台管理系统

---

## 7. Agent 能力边界

### 应该做的

- 理解用户的基础 Web3 数据查询意图
- 在需要时调用预定义工具
- 将工具结果整理成自然语言回复
- 保持最小上下文连续性
- 在高风险问题上保守回答

### 不应该做的

- 自行捏造链上数据
- 对价格走势做确定性承诺
- 提供明确买卖建议
- 在没有工具结果时假装自己拿到了实时数据

---

## 8. Web3 数据范围

### 允许接入的数据

- ETH 价格数据
- 地址 ETH 余额
- Gas 或基础 Token 信息

### 数据来源要求

- 数据来源必须可说明
- 链上数据与价格数据应明确区分
- 返回结果中应尽量体现“数据来自工具查询，而非模型主观生成”

---

## 9. 风险控制与免责声明

### 风险控制原则

- 所有高风险提问优先返回数据参考，不做操作建议
- 数据缺失、失败、超时时必须显式说明不确定性
- 工具失败时禁止模型伪造结果填补空白

### 免责声明原则

- 本项目是信息辅助工具，不构成投资建议
- 对市场判断、买卖时机、收益预测等请求，系统必须给出风险提示

---

## 10. 关键流程

### 主流程

1. 用户输入问题
2. Agent 判断是否需要调用工具
3. 若需要，调用对应 Web3 工具
4. 获取工具结果并回填给模型
5. 模型生成最终回复
6. 返回给前端，并保留当前会话上下文

### 异常流程

1. 工具参数无效
2. 工具执行失败
3. 外部 API 超时或返回异常
4. 用户提问超出系统能力边界

异常时必须：

- 返回可理解的失败说明
- 给出保守建议或下一步提示
- 不输出虚构数据

---

## 11. MVP 验收标准

- 用户可通过自然语言触发至少 2 个 Web3 工具
- 工具结果可用于生成最终自然语言回复
- 至少支持一次完整 Agent Loop
- 系统支持基础流式输出
- 会话可保留最小上下文
- 高风险问题会触发风险提示与免责声明
- 工具失败时不会输出伪造数据
- 文档体系足以指导后续实现

---

## 12. 后续迭代方向

MVP 之后可考虑的方向：

- RAG 接入协议文档和投研报告
- 多链支持
- Mock 交易工具
- 长期用户偏好 Memory
- 更完整的风险控制和审计能力

---

## 13. 当前结论

当前阶段不进入代码实现优先级争夺，先用文档、阶段说明和 skill 体系把项目边界、学习路径和开发流程固化下来。只有当 PRD、阶段执行说明和 skill 方案被确认后，才进入正式 vibe coding。
