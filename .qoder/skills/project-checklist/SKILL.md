# Project Checklist

## 作用

跟踪项目整体进度，明确已完成、未完成和未来计划的功能模块，为项目自我进化提供规划指导。

## 触发条件

- 用户执行 `/project-checklist` 命令
- 对话中包含关键词：更新 checklist、项目现状、项目进度、后续规划、未来计划
- 每次交付型任务完成后自动更新

## 输入

- 当前项目代码状态
- git 历史记录
- PRD 文档中的功能范围
- ARCHITECTURE 中的架构设计
- 最近的 changelog 记录

`docs/checklist/PROJECT-CHECKLIST.md`

## 流程

1. **收集项目状态**
   - 查看 git 历史确定已完成功能
   - 查看 PRD 确定 MVP 范围
   - 查看 ARCHITECTURE 确定架构设计
   - 查看 changelog 了解最近变更

2. **分类功能状态**
   - 已完成：代码已合并且测试通过
   - 进行中：已启动但未完成
   - 未完成：MVP 范围内但未启动
   - 未来规划：MVP 范围外的增强功能

3. **评估优先级**
   - P0（高）：核心功能、阻塞性问题、高价值
   - P1（中）：重要但不紧急、用户体验提升
   - P2（低）：锦上添花、长期优化

4. **生成文档**
   - 按标准模板填充内容
   - 计算完成率和关键指标
   - 给出下一步行动建议

5. **输出更新报告**
   ```markdown
   ## Checklist 更新摘要
   - 新增完成：X 项
   - 新增规划：X 项
   - 当前完成率：X%
   - 下一步建议：...
   ```

## 边界

- 不修改代码，仅更新文档
- 不重新定义需求范围（参考 PRD）
- 不替代 `/update-map`（checklist 关注功能清单，update-map 关注项目状态）
- 不替代 `/digest`（checklist 关注规划，digest 关注经验）

## 更新频率

- 交付型任务完成后自动更新
- 用户主动询问时更新
- 建议每周至少更新一次

## 规则

1. 必须基于实际代码状态，不虚构完成情况
2. 优先级评估需考虑价值和成本
3. 技术债务必须明确记录
4. 下一步建议必须可执行
5. 完成率计算必须准确
6. 保留历史更新记录（可选）
