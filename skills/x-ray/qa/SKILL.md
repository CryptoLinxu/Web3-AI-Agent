# QA

## 定位

QA 不是“最后顺手测一下”，而是负责定义和执行验证策略。

## 两种工作模式

### 1. RED 模式

适用：
- `FEAT`
- 需要先定义测试边界的复杂任务

目标：
- 在实现前先把测试写成红灯

规则：
1. 先写测试或验证清单
2. 先执行 RED
3. RED 只需要证明“当前还没通过”
4. RED 验证最多运行 2 次

### 2. VERIFY 模式

适用：
- `PATCH`
- `REFACTOR`
- 非 TDD 型补丁

目标：
- 验证修复和回归风险

## 输入

- `check-in`
- 架构说明
- 任务卡

## 输出

- 测试清单
- 红灯结果或验证结果
- 回归检查点

## 红绿规则

1. `FEAT` 先红后绿。
2. QA 阶段先负责 RED。
3. `coder` 阶段负责把 RED 全部变成 GREEN。
4. 如果 RED 直接意外通过，说明测试可能写弱了，需要修正。

## 边界

- 不直接写业务实现
- 不扩大需求范围

## 衔接

- `FEAT`：进入 `coder`
- `PATCH / REFACTOR`：进入 `coder` 或 `audit`

## 规则

1. QA 必须引用 `check-in` 的完成标准。
2. REFACTOR 默认优先保障回归验证。
3. PATCH 至少要保留轻量回归检查。
