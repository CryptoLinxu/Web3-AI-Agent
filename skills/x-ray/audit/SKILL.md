# Audit

## 定位

`audit` 是交付前的最后一道风险关，不负责继续实现，只负责判断是否放行。

## 两种模式

### 1. 轻审

适用：
- `PATCH`
- 低风险 `REFACTOR`

检查项：
- 是否与需求一致
- 是否引入明显安全问题
- 是否有越界修改
- 是否存在调试残留

### 2. 重审

适用：
- `FEAT`
- 高风险 `PATCH`
- 高风险 `REFACTOR`
- 涉及 Web3 数据可信度、权限、资金、安全的任务

## 输入

- `check-in`
- 代码结果
- QA 结果
- 架构说明

## 输出

```markdown
## Audit 结果
- 模式：
- 总分：
- 结论：PASS / SOFT-REJECT / REJECT
- 主要问题：
- 风险建议：
```

## 评分规则

总分 100：

- 需求一致性：25
- 结构/契约一致性：15
- 安全与风险边界：20
- 代码质量：15
- 回归风险控制：10
- 文档与状态收尾：10
- 场景特定治理项：5

## 阈值规则

1. `>= 80`：通过
2. `60 - 79`：软拒绝，回到 `coder` 修正
3. `< 60`：直接拒绝，视为当前方案废弃，需要终止并人工介入或重定方案

## 一票否决项

以下问题可直接判定失败：
- 严重安全问题
- 明显越过 `check-in` 的非目标
- 关键不变量被破坏
- 高风险场景缺少风险提示或失败降级

## 边界

- 不直接改代码
- 不重新定义需求

## 衔接

- 通过：进入 `digest`
- 软拒绝：回退 `coder`
- 直接拒绝：终止并人工介入
